# Contributing Guide

This document describes the procedures and rules for contributing to this repository (Issues, Pull Requests, etc.).  
This document does **not** replace the `LICENSE`. All usage and distribution rights of the source code are governed solely by the `LICENSE`.

> Important  
> This project is **not open source**.  
> While the source code is publicly available, building, redistributing, or distributing derivative products is not permitted.

---

## 1. Core Principles

External contributions are welcome, but the following principles apply:

- The **direction and ownership of the project are maintained** by the project owner.
- All contributions must have a **clear purpose and justification**.
- Changes that significantly increase maintenance cost may be rejected.

---

## 2. Scope of Contributions

The following types of contributions are welcome:

- Bug fixes
- Performance improvements
- Stability improvements
- Code readability improvements and refactoring (without behavioral changes)
- Documentation improvements (README, comments, etc.)
- Test additions or improvements
- Build and development environment improvements

The following changes **require prior discussion**:

- Changes to core systems, rules, or architecture
- Large dependency additions or replacements
- Features that significantly alter the project’s direction

In such cases, please open an Issue for discussion before submitting a Pull Request.

---

## 3. Legal: Contribution Rights and Ownership (Important)

By submitting a Pull Request or any other contribution, you agree that:

- You represent and warrant that you have the **legal right** to submit the contribution.
- The project owner may:
  - use,
  - modify,
  - merge,
  - redistribute,
  - relicense, and
  - commercially use  
  your contribution as part of the project.
- You waive any claim to **compensation, royalties, or usage restrictions** for your contribution.

This clause exists to protect the long-term development and commercial distribution of the project.

---

## 4. Third-Party Code and Assets

The following are **strictly prohibited**:

- Code with unclear or incompatible licensing
- Copying code from other projects without proper authorization
- Images, sounds, fonts, or data with unclear ownership
- External libraries whose licenses conflict with this project’s licensing policy

If your contribution includes AI-generated content, you must clearly state in the Pull Request description:

- The tool used
- The generation method
- The basis for determining usage and ownership rights

---

## 5. Pull Request Rules

Pull Requests must meet the following requirements:

- Each Pull Request should address **a single, focused purpose**.
- Changes and intent must be clearly explained.
- Include test methods or reproduction steps when applicable.
- **Do not include build artifacts** (executables, DLLs, ZIP files, large binaries, etc.).

Recommended PR description items:
- Summary of changes
- Reason for the change
- Scope of impact
- Testing status

---

## 6. Coding and Quality Standards

- Follow the language standards and conventions used by the project.
- Preserve existing code style as much as possible.
- Contributions must not introduce compilation errors or obvious warnings.
- Platform-specific code should be clearly isolated and documented.

Detailed rules may be added as the project evolves.

---

## 7. Security Issues

Security vulnerabilities or potential exploits should **not** be reported via public Issues.  
Please report them through a private communication channel instead.

Publicly posted security issues may be removed or made private without notice.

---

## 8. Communication and Conduct

- Contributors are expected to communicate respectfully.
- Aggressive or uncooperative behavior is not acceptable.
- Technical discussions should be grounded in evidence and context.

---

## 9. Final Notes

All contributions will be reviewed with appreciation; however,  
submission does not guarantee acceptance.

Decisions are made based on project stability, direction, and long-term maintainability.
